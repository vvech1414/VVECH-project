# 03 — Requirements

> Функциональные + нефункциональные требования к Patient App IntelDoc.

---

## 1. Functional Requirements

### FR-01 — Identity & Onboarding

**FR-01.1** Пользователь регистрируется через номер телефона + SMS-код
**FR-01.2** При первом входе пользователь проходит онбординг (3–5 экранов), объясняющий ценность приложения и consent-модель
**FR-01.3** Пользователь явно соглашается с обработкой персональных данных по 152-ФЗ перед первым использованием
**FR-01.4** Пользователь может настроить биометрический вход (Face ID / Touch ID) после регистрации
**FR-01.5** Поддерживается восстановление доступа через SMS на исходный номер

**Acceptance Criteria:**
- Время от старта приложения до завершения онбординга ≤ 90 секунд
- Все consent-экраны имеют чёткое "Согласен / Не согласен", без pre-selected опций
- При отказе от consent — приложение явно объясняет, что использование невозможно, и закрывается без сохранения данных

**Состояние в прототипе (2026-05-13):** реальной авторизации нет. Онбординг — 5 экранов в `src/routes/patient/entry/` (Welcome → Account → Access → Consents → Setup) + Vasily onboarding. Phone/SMS-flow смоделирован как UI-mock (без отправки SMS, без backend-валидации). Биометрия — UI-стаб. Состояние онбординга хранится в Zustand-флаге `hasCompletedOnboarding` (`src/store/store.ts`), который читается в `OnboardingGate` (`src/App.tsx:46`). **Статус: PARTIAL — UI готов, реальный auth отсутствует по scope.**

### FR-02 — Document Capture

**FR-02.1** Пользователь может загрузить документ через:
- Камеру смартфона (фото)
- Галерею (готовое фото / скан)
- Файловую систему (PDF)
**FR-02.2** При съёмке через камеру приложение автоматически кадрирует и выравнивает документ
**FR-02.3** Пользователь может загрузить multi-page документ (несколько фото в одну запись)
**FR-02.4** Поддерживаются форматы: JPG, PNG, PDF — заявлены в копирайте `src/copy/ru.ts` и UI флоу `UploadFlow.tsx` / `DocUpload.tsx`; фактической валидации MIME в прототипе нет (mock-загрузка).
**FR-02.5** Максимальный размер файла: в прототипе не enforced (нет реальной загрузки на backend; ограничение — браузерное).
**FR-02.6** При загрузке пользователь указывает тип документа (анализ, заключение, выписка, рецепт, другое)

**Acceptance Criteria:**
- Загрузка типичного документа (1–3 страницы) ≤ 10 секунд на 4G-сети
- При плохом качестве фото приложение предлагает перефотографировать
- Документ сохраняется локально на устройстве и синхронизируется в облако

### FR-03 — OCR & Structuring

**FR-03.1** Приложение автоматически распознаёт текст на загруженном документе
**FR-03.2** Для документов типа "анализ" — извлекает структурированные значения (показатель / значение / единица / норма)
**FR-03.3** Пользователь видит результат OCR и может его исправить вручную
**FR-03.4** Распознанные значения связываются с известным справочником показателей (TSH, HbA1c, глюкоза и т.д.)
**FR-03.5** При невозможности распознать — документ сохраняется как "не распознан", доступен в архиве

**Acceptance Criteria:**
- Точность OCR для типографских анализов ≥ 90% (на golden dataset из 50 документов)
- Время OCR одного документа ≤ 15 секунд
- При ручной правке пользователем — изменения логируются для аудита

**Состояние в прототипе:** OCR-engine — **симулированный** (`src/lib/ocr-mock.ts`, 53 строки). Возвращает детерминированные mock-результаты с разными confidence-уровнями (high / low / no-recognition) для демонстрации UI-состояний ревью и ручной правки. Никакого Tesseract / Vision API / отечественного движка в прототипе нет — это сознательное решение по scope (см. `intel-doc-prototype/CLAUDE.md` §3 "Minimal integration"). **Статус: PARTIAL — UI и состояния готовы, реальный OCR заменён mock'ом.**

---

### FR-04 — Document Library

**FR-04.1** Все загруженные документы отображаются в единой ленте, сортированной по дате
**FR-04.2** Доступны фильтры: по типу документа, по периоду, по статусу OCR
**FR-04.3** Доступен поиск по тексту распознанных документов
**FR-04.4** Каждый документ открывается в детальном виде с превью оригинала и структурированными данными
**FR-04.5** Пользователь может удалить документ (с подтверждением и логированием)
**FR-04.6** Пользователь может пометить документ как "важный" / "к следующему приёму"

**Acceptance Criteria:**
- Загрузка списка из 100+ документов ≤ 2 секунд
- Открытие детального view ≤ 1 секунды
- Удаление с возможностью отменить в течение 30 секунд (soft delete)

---

### FR-05 — Results & Trends

**FR-05.1** Для распознанных анализов приложение строит график значений показателя во времени
**FR-05.2** На графике обозначены границы нормы (если известны)
**FR-05.3** Значимые отклонения от тренда выделяются визуально
**FR-05.4** Пользователь может видеть тренд по нескольким показателям одновременно (compare view)
**FR-05.5** При недостатке данных (<2 измерений) показывается понятное сообщение, а не пустой график

**Acceptance Criteria:**
- График отображается за ≤ 1 секунду после открытия
- Хотя бы 5 ключевых эндокринных показателей имеют валидные диапазоны норм в справочнике
- При отсутствии данных нет fake-данных в demo-режиме

---

### FR-06 — Consent & Sharing

**FR-06.1** Пользователь может создать ссылку для передачи документа(ов) врачу
**FR-06.2** Перед созданием ссылки пользователь явно подтверждает: что передаётся, кому, на какой срок
**FR-06.3** Ссылка имеет TTL (24 часа / 7 дней / 30 дней — выбор пользователя)
**FR-06.4** Пользователь может отозвать ссылку в любой момент
**FR-06.5** Каждое открытие ссылки логируется (кто, когда)
**FR-06.6** Пользователь видит лог доступов к своим данным

**Acceptance Criteria:**
- Создание ссылки ≤ 5 секунд
- Отзыв ссылки делает её недоступной немедленно (< 1 секунда)
- Лог доступов показывает минимум: дата+время, получатель (если известен), действие

---

### FR-07 — Notifications

**FR-07.1** Push-уведомления при изменении статуса OCR (обработано)
**FR-07.2** Push-уведомления при открытии расшаренной ссылки врачом
**FR-07.3** Пользователь может настроить, какие типы уведомлений получать
**FR-07.4** Уведомления никогда не содержат медицинскую информацию в тексте (только нейтральные формулировки)

**Состояние в прототипе:** нативные push-уведомления **не реализованы** (нет Service Worker / Web Push, нет mobile-native). Вместо этого есть **in-app inbox** (`src/routes/patient/Notifications.tsx`) и страница действия по конкретному уведомлению (`NotificationAction.tsx`), плюс `NotificationBanner.tsx` для inline-баннеров. Push как канал доставки — TODO post-MVP. **Статус: PARTIAL — UI inbox готов, доставка вне приложения не реализована.**

---

### FR-08 — Profile & Settings

**FR-08.1** Пользователь видит и редактирует свой профиль (имя, дата рождения, контакты)
**FR-08.2** Пользователь может управлять активными consent-ами
**FR-08.3** Пользователь может скачать все свои данные (data portability по GDPR / 152-ФЗ)
**FR-08.4** Пользователь может удалить аккаунт со всеми данными
**FR-08.5** Доступна смена языка (RU / EN)

## 2. Non-Functional Requirements

### NFR-01 — Security

- **NFR-01.1** Данные на устройстве зашифрованы (AES-256)
- **NFR-01.2** Данные в передаче — TLS 1.3+
- **NFR-01.3** Серверная инфраструктура размещена на территории РФ (152-ФЗ требование)
- **NFR-01.4** Пароли никогда не хранятся в открытом виде (если применимо)
- **NFR-01.5** Биометрическая аутентификация использует системные API (Face ID / Touch ID), без хранения биометрии на серверах
- **NFR-01.6** Все запросы к API авторизованы JWT с коротким TTL (≤ 1 час)
- **NFR-01.7** Sensitive экраны (документы, результаты) не попадают в системные скриншоты / app-switcher preview

### NFR-02 — Privacy & Compliance

- **NFR-02.1** Соответствие 152-ФЗ "О персональных данных"
- **NFR-02.2** Соответствие ФЗ-242 (хранение данных граждан РФ на территории РФ)
- **NFR-02.3** Раздельные consent-ы на: обработку, передачу врачу, передачу клинике, рассылку
- **NFR-02.4** Возможность отзыва согласия в любой момент через интерфейс
- **NFR-02.5** Срок хранения данных после удаления аккаунта: не более 30 дней (с явным указанием в политике)
- **NFR-02.6** Audit log всех действий с данными хранится минимум 3 года

### NFR-03 — Performance

- **NFR-03.1** Время холодного старта приложения ≤ 3 секунд на устройствах от 2020 года
- **NFR-03.2** TTI (Time to Interactive) для главного экрана ≤ 2 секунд
- **NFR-03.3** Все основные действия (открытие документа, поиск, фильтр) ≤ 1 секунды
- **NFR-03.4** Приложение работает offline для просмотра уже загруженных документов
- **NFR-03.5** Размер APK / IPA ≤ 50 MB

### NFR-04 — Reliability

- **NFR-04.1** Crash-free sessions ≥ 99.5%
- **NFR-04.2** Backend uptime ≥ 99.9%
- **NFR-04.3** Бэкап данных пользователя — каждые 24 часа
- **NFR-04.4** При offline-загрузке документ сохраняется локально и синхронизируется при появлении сети

### NFR-05 — Usability & Accessibility

- **NFR-05.1** Соответствие WCAG 2.1 уровень AA
- **NFR-05.2** Поддержка screen readers (VoiceOver / TalkBack)
- **NFR-05.3** Возможность увеличить размер шрифта системно (Dynamic Type)
- **NFR-05.4** Минимальный контраст текста к фону: 4.5:1
- **NFR-05.5** Все интерактивные элементы — минимум 44×44 pt tap area
- **NFR-05.6** UI протестирован на пользователях 55+ (целевой сегмент эндо-пациентов)

### NFR-06 — Localization

- **NFR-06.1** Полная локализация на русский (primary)
- **NFR-06.2** Английский интерфейс — опционально на втором этапе
- **NFR-06.3** Форматы дат, чисел — по locale устройства
- **NFR-06.4** Медицинская терминология — на русском (международные сокращения дублируются)

### NFR-07 — Maintainability

- **NFR-07.1** Code coverage ключевых модулей (auth, document, consent) ≥ 70%
- **NFR-07.2** Все секреты — в env / secret manager, не в коде
- **NFR-07.3** Логирование на уровне ошибок и ключевых событий, без PII в логах
- **NFR-07.4** Документация компонентов в `docs/design-system/components.md`

### NFR-08 — Audit & Observability

- **NFR-08.1** Каждое действие пользователя над документом логируется (create, read, update, delete, share)
- **NFR-08.2** Лог содержит: timestamp, user_id, action, resource_id, IP / device_id
- **NFR-08.3** Лог immutable (append-only)
- **NFR-08.4** Пользователю доступен view собственного audit log через приложение

## 4. Open Questions

- Какие NFR валидируются автотестами в прототипе vs только описаны в документации? → **В прототипе автотестов нет** (нет `*.test.ts`, нет vitest/jest в `package.json`). Все NFR в защите идут как design-intent для production, не как verified-в-коде свойства. Это нужно открыто проговорить на защите.
- Нужна ли проработка edge cases для слабого интернета / низкосортных устройств для защиты или это nice-to-have? → На защите можно ограничиться demo на стабильном Wi-Fi; offline-mode UI-стабы уже есть, реальный Service Worker — post-MVP.
