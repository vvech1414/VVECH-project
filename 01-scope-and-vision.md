## 1. Scope защиты (что демонстрируем)

### In-Scope
- **Patient App** — пациентское React-приложение IntelDoc
- Сквозные пользовательские сценарии:
  - Онбординг и первичная аутентификация
  - Загрузка медицинских документов (фото / скан / PDF)
  - OCR-распознавание результатов анализов
  - Просмотр истории документов и результатов
  - Consent / согласие на обработку данных и передачу врачу
  - `[VERIFY_WITH_CODE]` — список фактических экранов в `src/`

### Out of Scope
- Doctor web dashboard (существует в полной версии продукта, но не в защите)
- Admin panel
- Реальная интеграция с МИС клиники
- Биллинг и оплата услуг
- Telemedicine / видеосвязь
- Аналитика и BI

### Boundary clarifications

- На публичной защите клиника-пилот фигурирует как "Клиника N" (анонимизация).
- Multi-language: основной — русский, опционально английский на интерфейсе.

## 2. Vision

### Problem Statement
Пациенты с хроническими эндокринными заболеваниями (диабет, заболевания щитовидной железы, гормональные нарушения) ведут длительное наблюдение, накапливая большое количество медицинских документов: результаты анализов, заключения врачей, выписки, рекомендации. Они вынуждены:
- Хранить бумажные документы и фотографии в разных местах
- Привозить пачки документов на каждый приём
- Объяснять врачу контекст с нуля при смене специалиста
- Не понимать тренд своих показателей между визитами
- Тревожиться из-за непонятных цифр в анализах

### Solution Vision
IntelDoc Patient App — **единый цифровой кабинет пациента** для эндокринологической клиники, который:
1. **Собирает** все медицинские документы пациента в одном месте через фото / скан / загрузку
2. **Понимает** содержание документов через OCR + структурирование данных
3. **Передаёт** структурированные данные лечащему врачу с явным consent на каждое действие
4. **Объясняет** результаты пациенту в понятной форме (контекст, тренд, норма)
5. **Соответствует** требованиям 152-ФЗ и медицинским стандартам обработки персональных данных

### Value Proposition
- **Для пациента:** меньше тревоги, лучше подготовлен к приёму, видит свою динамику
- **Для клиники:** более информированные приёмы, меньше времени на сбор анамнеза, лояльность пациентов
- **Для системы здравоохранения:** структурированные продольные данные о пациенте

### Ключевые дифференциаторы
1. **Специализация на эндокринологии** — не "ещё один общий медкабинет", а заточенный под продольное наблюдение
2. **OCR + структурирование** — не просто хранение PDF, а извлечение значений анализов с тегированием
3. **Audit-first архитектура** — каждое действие с данными логируется, что критично под 152-ФЗ
4. **Patient-centric consent** — пациент явно соглашается на каждую передачу данных, не "галочкой в EULA"

## 4. Stakeholders

### Primary
- **Пациент** (Persona: Анна, 45 лет, диабет 2 типа, использует смартфон ежедневно) — основной пользователь
- **Лечащий врач** (Persona: Сергей, эндокринолог в клинике) — получатель данных, **не пользователь Patient App**, но дизайн должен учитывать его сценарий приёма

### Secondary
- **Регистратура / администратор клиники** — записывает пациента, видит подтверждение consent
- **DPO / специалист по защите данных клиники** — аудит compliance с 152-ФЗ
- **IT-команда клиники** — интеграция с МИС

### External
- **Регулятор (Роскомнадзор, Минздрав)** — compliance
- **Команда защиты в образовательной программе** — приёмка проекта

## 5. E-State / To-B-State

### E-State (как сейчас работает пациент)
- Бумажные документы хранятся в папках дома
- Фотографии анализов разбросаны по галерее смартфона
- На приём пациент приносит "охапку бумаг"
- Между приёмами нет понимания динамики
- При смене врача — полная пересдача анамнеза

### To-B-State (после внедрения IntelDoc Patient App)
- Все документы в одном приложении, структурированные по типу и дате
- На приём врач получает заранее систематизированные данные (с consent)
- Пациент видит тренды ключевых показателей
- При смене врача — клик "поделиться" + явное consent
