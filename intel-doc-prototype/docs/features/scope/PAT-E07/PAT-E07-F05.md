# PAT-E07-F05 — Объединённая лента истории

> Эпик: **PAT-E07** История анализов и карточка анализа
> Поверхность: Приложение пациента (PAT) · Путешествие: Ежедневная подготовка пациента (J2)

## 1. Что делает фича

- **Шаг:** 5
- **Экран:** Объединённая лента истории (`/patient/history`, 4-я вкладка таб-бара)
- **Действия пользователя:** Открывает вкладку «История», просматривает хронологию событий, тапает карточку анализа или запроса врача для перехода к деталям, либо FAB для добавления нового анализа
- **Данные системы:** `analyses[]`, `documents[]`, `accessGrants[]`, `doctorRequests[]`, `appointment`, `overduePlanItems[]` — собираются в единую ленту, отсортированную по timestamp DESC, и сгруппированную по календарному дню в локальной TZ
- **Happy path:** Видит ленту событий, сгруппированных по дням, с цветовой кодировкой по типу и статус-бейджами; тапает анализ → попадает в карточку анализа (PAT-E07-F01)
- **JTBD:** Иметь одно место, где видна вся хронология взаимодействия с клиникой — что загружено, кому открыт доступ, какие запросы пришли от врача, какие приёмы запланированы

## 2. Контекст эпика

- **Initiative flow:** Пациент открывает историю или карточку → видит документ и структурированные значения → понимает что уже доступно врачу
- **JTBD functional:** Видеть, что уже загружено и доступно врачу
- **JTBD emotional:** Доверие, что данные не потерялись и взаимодействие с клиникой прозрачно
- **Состояние эпика в прототипе:** Готово (базовая лента) / В работе (guardrail-состояния)
- **Комментарий из реестра:** Расширение F02 «Список истории с фильтрами» до многотипового агрегатора. Не заменяет F02 — F02 остаётся фильтрованным видом анализов; F05 — таб-уровневый агрегатор всех долгоживущих событий

## 3. Применимые гардрейлы

- **Прототип, не продакшн** — оптимизация под ясность демо, mock-state, без реальных API/OCR/auth/БД.
- **AI consultative only** — никаких diagnosis/treatment/prescription формулировок; разрешены summary, clarification, prep-guidance, neutral explanations.
- **Прозрачность доступа** — пациент видит recipient, scope, срок, статус доступа; никакой «тихой» передачи данных. В ленте отдельно видны события `выдан / отозван / истёк`.
- **Партнёр-first UX** — ЭНЦ-контекст виден в карточках доступа («Доступ выдан клинике ЭНЦ»), ощущение клинико-связанного канала.
- **A11y и возраст 35–65** — крупные tap-targets (карточки 56+ px высоты), читаемый шрифт (15 px на title), низкая когнитивная нагрузка (один акцент на карточку — иконка + бейдж).
- **Copy на русском** — короткие лейблы, supportive helper-тексты, без алармизма.
- **Reusable components** — использовать компоненты из `src/components/` (см. §6).

## 4. Применимые NFR

- Лента собирается из единого Zustand-store без локального дубля состояния — изменения в любом другом флоу (загрузка анализа, отзыв доступа, новый запрос) автоматически отражаются здесь
- Сортировка и группировка — на клиенте, без асинхронных запросов
- Скролл плавный на iPhone 390×844; FAB не перекрывает последний элемент списка (нижний padding ≥ 110 px)

## 5. Состояния экрана (state coverage)

- **default** — Лента с событиями, сгруппированными по дням. ★ обязательно для демо
- **loading** — (не применимо — данные локальные, рендер мгновенный)
- **empty** — Пациент ещё ничего не загружал и не получал событий. Показывается illustration + копирайт + CTA `Добавить анализ`. FAB скрыт, чтобы не дублировать CTA.
- **error** — (не применимо к прототипу)
- **success** — Применимо к точечным переходам: после возврата из загрузки анализа в ленту видно новое событие сверху

**Дополнительные guardrail-состояния, обязательные к показу:**
- `Требует проверки` — анализ с low-confidence OCR (warning-бейдж на карточке)
- `Истёк` / `Отозван` — карточки события доступа в негативном финале
- `Просрочено` — карточка просроченного пункта плана врача
- `Новое` — карточка непросмотренного запроса врача (warning-бейдж + точка-индикатор)

> ★ Обязательно показать в демо: **default** + минимум один guardrail-бейдж в ленте Марии Ивановой

## 7. Mock-данные

```ts

interface Event {
  id: string
  kind: EventKind
  Icon: LucideIcon
  title: string
  description?: string
  timestamp: string  // ISO
  href?: string      // переход на детальный экран, если применимо
```

## 8. Copy guidance

**Тон:** спокойный, практичный, не алармистский. Короткие лейблы, поддерживающий helper-текст.

**Заголовок экрана:** `История`

**Empty state:**
- Title: `Здесь будет история ваших действий`
- Body: `Когда вы начнёте загружать анализы и получать обновления — они появятся здесь.`
- CTA: `Добавить анализ`

**Карточки по типам:**
- Анализ: `{label}` / `Распознано и сохранено` или `Принято врачом` · бейджи `Загружено` / `Принято` / `Требует проверки`
- Документ: `Документ: {label}` / `Загружен и зашифрован` · бейдж `Готово`
- Доступ: `Доступ выдан клинике ЭНЦ` / `Бессрочно, для всей клиники` · `Доступ отозван` (бейдж `Отозван`) · `Доступ истёк` (бейдж `Истёк`)
- Запрос: `Запрос от врача` / `{title}` · бейджи `Новое` / `Просмотрено`
- Приём: `Приём запланирован` / `{дата, время}` · бейдж `Запись`
- Просрочка: `{label пункта}` / `Просрочен пункт плана` · бейдж `Просрочено`

**Заголовок дня (caps, ink-muted):** `12 апреля`, `11 апреля`, ...

**Избегать:**
- «диагноз»
- «назначение лечения»
- «рецепт» как ключевая функция AI/app
- алармистские формулировки

## 10. Открытые вопросы

- Нужно ли в ленте отдельно показывать события «приём подтверждён» / «приём отменён» помимо «приём запланирован»? Сейчас в store у `Appointment` только `scheduled` / `completed`.
- Стоит ли группировать однотипные события одного дня в один аккордеон (например, 3 загрузки документов подряд) или оставить плоский список? Для прототипа — плоский, но в проде это станет шумно.
